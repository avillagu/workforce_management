# PROMPT PARA AGENTE: BACKEND DEVELOPER - ETAPA 3 (PROGRAMACIÓN)

## ROL
Eres el Backend Developer del sistema MAPO, experto en Node.js, Express y PostgreSQL.

## OBJETIVO
Construir la API RESTful completa para la gestión de turnos (Programación) con soporte para tiempo real.

## CONTEXTO
- Proyecto: WFM - Etapa 3: Programación de Turnos
- Backend existente: Node.js/Express con Swagger en `/api-docs`
- Base de datos: PostgreSQL con esquema `wfm_auth`
- Tabla `turnos` ya definida en `architecture_erd.md`
- Autenticación: JWT con middleware existente
- Librería de fechas: luxon (ya instalada)

## TAREAS PRINCIPALES

### 1. CRUD de Turnos
Crear rutas en `backend/src/routes/turnos.js` y controlador `turnosController.js`:

- `GET /api/turnos?fecha_inicio=YYYY-MM-DD&fecha_fin=YYYY-MM-DD`
  * Retorna todos los turnos en el rango de fechas
  * Incluir datos del empleado (nombre, username) mediante JOIN con usuarios
  
- `GET /api/turnos/empleado/:usuario_id?fecha_inicio=&fecha_fin=`
  * Retorna turnos de un empleado específico en rango de fechas
  
- `POST /api/turnos`
  * Crea un turno individual
  * Body: `{ usuario_id, hora_inicio, hora_fin }`
  * Validar que no exista turno superpuesto para el mismo empleado
  * Hash de fechas con luxon en UTC
  
- `PUT /api/turnos/:id`
  * Actualiza un turno existente
  * Validar superposiciones excluyendo el turno actual
  
- `DELETE /api/turnos/:id`
  * Elimina un turno (físicamente o lógicamente según ERD)

### 2. Programación Masiva
- `POST /api/turnos/masivo`
  * Body: `{ empleado_ids: string[], fecha_inicio: string, fecha_fin: string, hora_inicio: string, hora_fin: string }`
  * Crea turnos para múltiples empleados en un rango de fechas
  * Ejemplo: Programar turno 08:00-17:00 para 5 empleados durante 1 semana
  * Validar conflictos por cada empleado
  * Transacción de base de datos (todo o nada)

### 3. Validación de Conflictos
- Implementar función `validarSuperposicion(usuario_id, hora_inicio, hora_fin, turno_id_excluir = null)`
- Un turno superpone si:
  * Mismo empleado
  * Mismo día
  * Las horas se solapan (inicio_nuevo < fin_existente Y fin_nuevo > inicio_existente)
- Retornar error 409 Conflict con detalles si hay superposición

### 4. Tiempo Real (WebSocket)
- Implementar Socket.IO en el backend
- Configurar evento `turnos:actualizado` que se emita cuando:
  * Se crea un turno
  * Se actualiza un turno
  * Se elimina un turno
- El payload debe incluir: `{ accion: 'crear'|'actualizar'|'eliminar', turno: {...} }`

### 5. Swagger Documentation
- Documentar TODOS los endpoints con JSDoc/Swagger
- Incluir ejemplos de request/response
- Especificar códigos de error (400, 404, 409)

### 6. Manejo de Errores
- 400: Datos inválidos o faltantes
- 404: Turno no encontrado
- 409: Conflicto de superposición
- 500: Error interno con logging

## RESTRICCIONES CRÍTICAS

1. **Fechas UTC**: Usar `luxon` y `DateTime.utc()`. Todas las fechas se guardan en UTC.
2. **UUIDs**: Usar `crypto.randomUUID()` para IDs.
3. **Consultas preparadas**: Prevenir SQL injection con `$1, $2`.
4. **Middleware auth**: Proteger todos los endpoints con el middleware JWT existente.
5. **Roles**: 
   - GET: admin, supervisor, empleado (cada uno ve sus datos según rol)
   - POST/PUT/DELETE: solo admin
6. **No inventar columnas**: Revisar estrictamente `architecture_erd.md`.
7. **Transacciones**: Usar `BEGIN/COMMIT/ROLLBACK` en programación masiva.
